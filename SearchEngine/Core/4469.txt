{"text": "Language Compiler and Runtime System Support for Highly Scalable Big Data Systems About Modern computing has entered the era of Big Data Developing systems that can scale to massive amounts of data with relatively small amounts of resources is a key challenge faced by both researchers and practitioners The mainstream approach to scalability is to enable distributed data processing As a result existing platforms utilize large numbers of machines in clusters or in the cloud data are partitioned among machines so that many processors can work simultaneously on a task Most of the existing Big Data systems are written in managed Languages such as Java C or Scala which are known for their simple usage easy memory management and large community support While these languages simplify development effort their managed runtime has a high cost which cannot be amortized by increasing the number of data processing machines in a cluster Poor performance on each node reduces the scalability of the entire cluster a large number of machines are needed to process a small dataset resulting in excessive use of resources and increased communication overhead This project explores a new direction to scale Big Data systems that is how to effectively optimize the managed runtime of a data processing system to improve its performance and scalability on each machine The current focus of our work is in the following three directions 1 Facade A compiler and runtime system for almost object bounded Big Data applications A managed Big Data application often suffers from large space overhead and GC cost due to extremely large numbers of objects and references in the heap A key observation is that in a scalable system the number of heap objects representing data items cannot grow proportionally with the dataset cardinality We develop Facade a Java based compiler and runtime that can statically bound the number of heap objects that represent data items Facade advocates to store data items in native memory and create objects as facades to represent data items It uses a new execution model that dynamically establishes a many to one mapping between an unbounded set of data items in native memory and a statically bounded set of objects in the heap thereby reducing significantly the number of objects their associated space overhead i e pointers and headers as well as the GC cost Please read this technical report for details 2 ScaleJ A language and autotuner for scalable and efficient data processing A common performance problem identified in our study is the overly parallel execution of data processing tasks because data processing threads often work on distinct data partitions and have very few dependencies between each other developers tend to create a large number of them to fully utilize the underlying parallelism However the excessive creation of objects and use of object oriented data structures incurs tremendous runtime bloat a term referring to overly complex computations that unnecessarily hurt performance that is duplicated for each thread created making the system quickly hit the memory wall on a shared memory architecture On the other hand using a very small number of threads e g one will guarantee scalability i e no out of memory error but at the cost of sacrificing processing speed Statically determining the appropriate number of threads can be extremely difficult since it is often a result of dynamic interplay between multiple factors such as the size of input data the amount of available memory and the space overhead of the data structures e g collections used We propose a new language ScaleJ that enables data driven adjustment of parallelism DDAP the execution always starts with a large number of threads to maximize speed Our compiler support provides instrumentation that does lightweight profiling to understand the memory usage in each iteration Based on the profiling results our autotuning system periodically predicts whether the available memory is enough to support the rest of the iterations If not the number of threads needs to be reduced to lower the memory pressure in the system DDAP provides the following scalability guarantee as long as a dataset can be successfully processed by one single thread our technique will find the parallelism sweetspot at which the program is guaranteed to run successfully with the processing speed maximized The project is currently in progress We will report more detailed results here when they are ready 3 Semantics aware graph simplification Real world graphs such as the Yahoo webgraph and the twitter graph are extremely large and often need a cluster of machines to process To support efficient test and debugging it is important to generate from real world graphs a small subset of vertices and edges that retain interesting properties in the original graphs Existing techniques focused primarily on graph sampling that uses statistical methods to sample a large graph so that the generated graph follows the same distribution However all of these techniques are semantics agnostic meaning that they prune graph without considering what each application is interested in For example page rank is interested in edge density while maximal clique cares more about the sizes of the cliques in the graph We are in the process of developing novel algorithms that intelligently prune a graph based on the user s specifications of interesting properties More details will be reported here Papers o A bloat aware design for Big Data Applications Support for Highly Scalable Big Data Systems</span><span\r\nstyle='mso-fareast-font-family:\"Times New Roman\"'><o:p></o:p></span></h1>\r\n\r\n<div class=MsoNormal align=center align=center>\r\n\r\n</span></div>\r\n\r\n<p class=MsoNormal style='mso-layout-grid-align:none;text-autospace:none'><b><span\r\nstyle='font-size:24.0pt;mso-fareast-font-family:\"Times New Roman\";color:black;\r\nmso-font-kerning:18.0pt'>About<o:p></o:p></span></b></p>\r\n\r\n<p class=MsoNormal style='mso-layout-grid-align:none;text-autospace:none'><span\r\nstyle='mso-fareast-font-family:\"Times New Roman\";color:black'><o:p>&nbsp;</o:p></span></p>\r\n\r\n<p class=MsoNormal style='mso-layout-grid-align:none;text-autospace:none'><span\r\nstyle='mso-fareast-font-family:\"Times New Roman\";color:black'>Modern computing\r\nhas entered the era of Big Data. Developing systems that can scale to massive\r\namounts of data with relatively small amounts of resources is a key challenge\r\nfaced by both researchers and practitioners. The mainstream approach to\r\nscalability is to enable distributed data processing. As a result, existing\r\nplatforms utilize large numbers of machines in clusters or in the cloud; data\r\nare partitioned among machines so that many processors can work simultaneously\r\non a task. </span><span style='font-size:11.0pt;font-family:\"NimbusRomNo9L-Regu\",\"serif\";\r\nmso-fareast-font-family:\"Times New Roman\";mso-bidi-font-family:NimbusRomNo9L-Regu'>Most\r\nof </span><span style='mso-fareast-font-family:\"Times New Roman\";color:black'>the\r\nexisting Big Data systems are written in managed Languages, such as Java, C#,\r\nor Scala, which are known for their simple usage, easy memory management, and\r\nlarge community support. While these languages simplify development effort,\r\ntheir managed runtime has a high cost, which cannot be amortized by increasing\r\nthe number of data-processing machines in a cluster. Poor performance on each\r\nnode reduces the scalability of the entire cluster: a large number of machines\r\nare needed to process a small dataset, resulting in excessive use of resources\r\nand increased communication overhead. This project explores a new direction to\r\nscale Big Data systems, that is, how to effectively optimize the managed\r\nruntime of a data processing system to improve its performance and scalability\r\non each machine. The current focus of our work is in the following three directions:<o:p></o:p></span></p>\r\n\r\n<p class=MsoNormal style='mso-layout-grid-align:none;text-autospace:none'><span\r\nstyle='mso-fareast-font-family:\"Times New Roman\";color:black'><o:p>&nbsp;</o:p></span></p>\r\n\r\n<p class=MsoNormal style='mso-layout-grid-align:none;text-autospace:none'><b\r\nstyle='mso-bidi-font-weight:normal'><span style='mso-fareast-font-family:\"Times New Roman\";\r\ncolor:black'>(1) Facade: A compiler and runtime system for (almost)\r\nobject-bounded Big Data applications<o:p></o:p></span></b></p>\r\n\r\n<p class=MsoNormal style='margin-left:.5in;mso-layout-grid-align:none;\r\ntext-autospace:none'><span style='mso-fareast-font-family:\"Times New Roman\";\r\ncolor:black'>A managed Big Data application often suffers from large space\r\noverhead and GC cost due to extremely large numbers of objects and references\r\nin the heap. A key observation is that, in a scalable system, the number of\r\nheap objects representing data items cannot grow proportionally with the\r\ndataset cardinality. We develop Facade, a Java-based compiler and <span\r\nclass=GramE>runtime, that</span> can statically bound the number of heap\r\nobjects that represent data items. Facade advocates <span class=GramE>to store\r\ndata items in native memory and create</span> objects as facades to represent\r\ndata items. It uses a new execution model that dynamically establishes a\r\nmany-to-one mapping between an unbounded set of data items in native memory and\r\na statically bounded set of objects in the heap, thereby reducing significantly\r\nthe number of objects, their associated space overhead (i.e., pointers and\r\nheaders), as well as the GC cost. Please read this <a\r\nhref=\"../../papers/facade-tr-14.pdf\">technical report</a> for details.<o:p></o:p></span></p>\r\n\r\n<p class=MsoNormal style='mso-layout-grid-align:none;text-autospace:none'><span\r\nstyle='mso-fareast-font-family:\"Times New Roman\";color:black'><o:p>&nbsp;</o:p></span></p>\r\n\r\n<p class=MsoNormal><b><span style='mso-fareast-font-family:\"Times New Roman\";\r\ncolor:black'>(2) <span class=SpellE>ScaleJ</span>: A language and <span\r\nclass=SpellE>autotuner</span> for scalable and efficient data processing </span></b><span\r\nstyle='mso-fareast-font-family:\"Times New Roman\";color:black'><o:p></o:p></span></p>\r\n\r\n<p class=MsoNormal style='margin-left:.5in;mso-layout-grid-align:none;\r\ntext-autospace:none'><span style='mso-fareast-font-family:\"Times New Roman\";\r\ncolor:black'>A common performance problem identified in our study is the\r\noverly-parallel execution of data processing tasks: because data processing\r\nthreads often work on distinct data partitions and have very few dependencies\r\nbetween each other, developers tend to create a large number of them to fully\r\nutilize the underlying parallelism. However, the excessive creation of objects\r\nand use of object-oriented data structures incurs tremendous runtime bloat (a\r\nterm referring to overly complex computations that unnecessarily hurt\r\nperformance that is duplicated for each thread created, making the system\r\nquickly hit the memory wall on a shared memory architecture. On the other hand,\r\nusing a very small number of threads (e.g., one) will guarantee scalability\r\n(i.e., no out-of-memory error), but at the cost of sacrificing processing\r\nspeed. Statically determining the appropriate number of threads can be\r\nextremely difficult, since it is often a result of dynamic interplay between\r\nmultiple factors, such as the size of input data, the amount of available\r\nmemory, and the space overhead of the data structures (e.g., collections) used.\r\nWe propose a new language <span class=SpellE>ScaleJ</span> that enables </span><i><span\r\nstyle='mso-fareast-font-family:\"Times New Roman\"'>data-driven adjustment of\r\nparallelism </span></i><span style='mso-fareast-font-family:\"Times New Roman\"'>(DDAP):\r\nthe execution always starts with a large number of threads to maximize speed. <span\r\nstyle='color:black'>Our compiler support provides instrumentation that does\r\nlightweight profiling to understand the memory usage <span class=GramE>in each\r\niteration</span>. Based on the profiling results, our <span class=SpellE>autotuning</span>\r\nsystem periodically predicts whether the available memory is enough to support\r\nthe rest of the iterations. If not, the number of threads needs to be reduced\r\nto lower the memory pressure in the system. </span>DDAP provides the following\r\nscalability guarantee: <i>as long as a dataset can be successfully processed by\r\none single thread, our technique will find the parallelism <span class=SpellE>sweetspot</span>\r\nat which the program is guaranteed to run successfully with the processing\r\nspeed maximized</i>. The project is currently in progress. To support\r\nefficient test and debugging, it is important to generate, from real-world\r\ngraphs, a small subset of vertices and edges that retain interesting properties\r\nin the original graphs. Existing techniques focused primarily on graph sampling\r\nthat uses statistical methods to sample a large graph so that the generated\r\ngraph follows the same distribution. However, all of these techniques are\r\nsemantics-agnostic, meaning that they prune graph without considering what each\r\napplication is interested in. For example, page rank is interested in edge\r\ndensity while maximal clique cares more about the sizes of the cliques in the\r\ngraph.
