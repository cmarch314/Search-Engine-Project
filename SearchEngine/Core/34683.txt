Clicking the same header again reverses the order.\n</p>\n<h2 id=\"ChangingReportNumbering\">Changing Report Numbering</h2>\n<p>\nThere may be instances where you need to change the ID of the report, perhaps to organize the reports better. At present this requires changes to the trac database. The <em>report</em> table has the following schema <em>(since 0.10)</em>:\n</p>\n<ul><li>id integer PRIMARY KEY\n</li><li>author text\n</li><li>title text\n</li><li>query text\n</li><li>description text\n</li></ul><p>\nChanging the ID changes the shown order and number in the <em>Available Reports</em> list and the report's perma-link. This is done by running something like:\n</p>\n<pre class=\"wiki\">update report set id=5 where id=3;\n</pre><p>\nKeep in mind that the integrity has to be maintained (i.e., ID has to be unique, and you don't want to exceed the max, since that's managed by SQLite someplace).\n</p>\n<p>\nYou may also need to update or remove the report number stored in the report or query.\n</p>\n<h2 id=\"NavigatingTickets\">Navigating Tickets</h2>\n<p>\nClicking on one of the report results will take you to that ticket. You can navigate through the results by clicking the <em>Next Ticket</em> or <em>Previous Ticket</em> links just below the main menu bar, or click the <em>Back to Report</em> link to return to the report page.\n</p>\n<p>\nYou can safely edit any of the tickets and continue to navigate through the results using the <em>Next/Previous/Back to Report</em> links after saving your results, but when you return to the report, there will be no hint about what has changed, as would happen if you were navigating a list of tickets obtained from a query (see <a class=\"wiki\" href=\"/LUCICodeRepository/nomaticIM/wiki/TracQuery#NavigatingTickets\">TracQuery#NavigatingTickets</a>). <em>(since 0.11)</em>\n</p>\n<h2 id=\"AlternativeDownloadFormats\">Alternative Download Formats</h2>\n<p>\nAside from the default HTML view, reports can also be exported in a number of alternative formats.\nAt the bottom of the report page, you will find a list of available data formats. Click the desired link to \ndownload the alternative report format.\n</p>\n<h3 id=\"Comma-delimited-CSVCommaSeparatedValues\">Comma-delimited - CSV (Comma Separated Values)</h3>\n<p>\nExport the report as plain text, each row on its own line, columns separated by a single comma (',').\n<strong>Note:</strong> The output is fully escaped so carriage returns, line feeds, and commas will be preserved in the output.\n</p>\n<h3 id=\"Tab-delimited\">Tab-delimited</h3>\n<p>\nLike above, but uses tabs (\\t) instead of comma.\n</p>\n<h3 id=\"RSS-XMLContentSyndication\">RSS - XML Content Syndication</h3>\n<p>\nAll reports support syndication using XML/RSS 2.0. To subscribe to an RSS feed, click the orange 'XML' icon at the bottom of the page. See <a class=\"wiki\" href=\"/LUCICodeRepository/nomaticIM/wiki/TracRss\">TracRss</a> for general information on RSS support in Trac.\n</p>\n<hr />\n<h2 id=\"CreatingCustomReports\">Creating Custom Reports</h2>\n<p>\n<em>Creating a custom report requires a comfortable knowledge of SQL.</em>\n</p>\n<p>\nA report is basically a single named SQL query, executed and presented by\nTrac.  Reports can be viewed and created from a custom SQL expression directly\nin the web interface.\n</p>\n<p>\nTypically, a report consists of a SELECT-expression from the 'ticket' table,\nusing the available columns and sorting the way you want it.\n</p>\n<h2 id=\"Ticketcolumns\">Ticket columns</h2>\n<p>\nThe <em>ticket</em> table has the following columns:\n</p>\n<ul><li>id\n</li><li>type\n</li><li>time\n</li><li>changetime\n</li><li>component\n</li><li>severity  \n</li><li>priority \n</li><li>owner\n</li><li>reporter\n</li><li>cc\n</li><li>version\n</li><li>milestone\n</li><li>status\n</li><li>resolution\n</li><li>summary\n</li><li>description\n</li><li>keywords\n</li></ul><p>\nSee <a class=\"wiki\" href=\"/LUCICodeRepository/nomaticIM/wiki/TracTickets\">TracTickets</a> for a detailed description of the column fields.\n</p>\n<p>\nExample: <strong>All active tickets, sorted by priority and time</strong>\n</p>\n<pre class=\"wiki\">SELECT id AS ticket, status, severity, priority, owner, \n       time AS created, summary FROM ticket \n  WHERE status IN ('new', 'assigned', 'reopened')\n  ORDER BY priority, time\n</pre><p>\n---\n</p>\n<h2 id=\"AdvancedReports:DynamicVariables\">Advanced Reports: Dynamic Variables</h2>\n<p>\nFor more flexible reports, Trac supports the use of <em>dynamic variables</em> in report SQL statements. \nIn short, dynamic variables are <em>special</em> strings that are replaced by custom data before query execution.\n</p>\n<h3 id=\"UsingVariablesinaQuery\">Using Variables in a Query</h3>\n<p>\nThe syntax for dynamic variables is simple, any upper case word beginning with '$' is considered a variable.\n</p>\n<p>\nExample:\n</p>\n<pre class=\"wiki\">SELECT id AS ticket,summary FROM ticket WHERE priority=$PRIORITY\n</pre><p>\nTo assign a value to $PRIORITY when viewing the report, you must define it as an argument in the report URL, leaving out the leading '$'.\n</p>\n<p>\nExample:\n</p>\n<pre class=\"wiki\"> http://trac.edgewall.org/reports/14?PRIORITY=high\n</pre><p>\nTo use multiple variables, separate them with an '&amp;'.\n</p>\n<p>\nExample:\n</p>\n<pre class=\"wiki\"> http://trac.edgewall.org/reports/14?PRIORITY=high&amp;SEVERITY=critical\n</pre><h3 id=\"SpecialConstantVariables\">Special/Constant Variables</h3>\n<p>\nThere is one dynamic variable whose value is set automatically (the URL does not have to be changed) to allow practical reports. \n</p>\n<ul><li>$USER \u2014 Username of logged in user.\n</li></ul><p>\nExample (<em>List all tickets assigned to me</em>):\n</p>\n<pre class=\"wiki\">SELECT id AS ticket,summary FROM ticket WHERE owner=$USER\n</pre><hr />\n<h2 id=\"AdvancedReports:CustomFormatting\">Advanced Reports: Custom Formatting</h2>\n<p>\nTrac is also capable of more advanced reports, including custom layouts,\nresult grouping and user-defined CSS styles. To create such reports, we'll use\nspecialized SQL statements to control the output of the Trac report engine.\n</p>\n<h2 id=\"SpecialColumns\">Special Columns</h2>\n<p>\nTo format reports, <a class=\"wiki\" href=\"/LUCICodeRepository/nomaticIM/wiki/TracReports\">TracReports</a> looks for 'magic' column names in the query\nresult. These 'magic' names are processed and affect the layout and style of the \nfinal report.\n</p>\n<h3 id=\"Automaticallyformattedcolumns\">Automatically formatted columns</h3>\n<ul><li><strong>ticket</strong> \u2014 Ticket ID number. Becomes a hyperlink to that ticket. \n</li><li><strong>id</strong> \u2014 same as <strong>ticket</strong> above when <strong>realm</strong> is not set\n</li><li><strong>realm</strong> \u2014 together with <strong>id</strong>, can be used to create links to other resources than tickets (e.g. a realm of <em>wiki</em> and an <em>id</em> to a page name will create a link to that wiki page)\n</li><li><strong>created, modified, date, time</strong> \u2014 Format cell as a date and/or time.\n</li><li><strong>description</strong> \u2014 Ticket description field, parsed through the wiki engine.\n</li></ul><p>\n<strong>Example:</strong>\n</p>\n<pre class=\"wiki\">SELECT id AS ticket, created, status, summary FROM ticket \n</pre><p>\nThose columns can also be defined but marked as hidden, see <a class=\"wiki\" href=\"/LUCICodeRepository/nomaticIM/wiki/TracReports#column-syntax\">below</a>.\n</p>\n<p>\nSee <a class=\"ext-link\" href=\"http://trac.edgewall.org/intertrac/wiki/CookBook/Configuration/Reports\" title=\"wiki/CookBook/Configuration/Reports in Trac project trac\"><span class=\"icon\">\u200b</span>trac:wiki/CookBook/Configuration/Reports</a> for some example of creating reports for realms other than <em>ticket</em>.\n</p>\n<h3 id=\"Customformattingcolumns\">Custom formatting columns</h3>\n<p>\nColumns whose names begin and end with 2 underscores (Example: <strong><tt>__color__</tt></strong>) are\nassumed to be <em>formatting hints</em>, affecting the appearance of the row.\n \n</p>\n<ul><li><strong><tt>__group__</tt></strong> \u2014 Group results based on values in this column. Each group will have its own header and table.\n</li><li><strong><tt>__grouplink__</tt></strong> \u2014 Make the header of each group a link to the specified URL. The URL is taken from the first row of each group.\n</li><li><strong><tt>__color__</tt></strong> \u2014 Should be a numeric value ranging from 1 to 5 to select a pre-defined row color. Typically used to color rows by issue priority.\n<div style=\"margin-left:7.5em\">Defaults: \n<span style=\"border: none; color: #333; background: transparent;  font-size: 85%; background: #fdc; border-color: #e88; color: #a22\">Color 1</span>\n<span style=\"border: none; color: #333; background: transparent;  font-size: 85%; background: #ffb; border-color: #eea; color: #880\">Color 2</span>\n<span style=\"border: none; color: #333; background: transparent;  font-size: 85%; background: #fbfbfb; border-color: #ddd; color: #444\">Color 3</span>\n<span style=\"border: none; color: #333; background: transparent; font-size: 85%; background: #e7ffff; border-color: #cee; color: #099\">Color 4</span>\n<span style=\"border: none; color: #333; background: transparent;  font-size: 85%; background: #e7eeff; border-color: #cde; color: #469\">Color 5</span>\n</div>\n</li><li><strong><tt>__style__</tt></strong> \u2014 A custom CSS style expression to use for the current row. \n</li></ul><p>\n<strong>Example:</strong> <em>List active tickets, grouped by milestone, group header linked to milestone page, colored by priority</em>\n</p>\n<pre class=\"wiki\">SELECT p.value AS __color__,\n     t.milestone AS __group__,\n     '../milestone/' || t.milestone AS __grouplink__,\n     (CASE owner WHEN 'daniel' THEN 'font-weight: bold; background: red;' ELSE '' END) AS __style__,\n       t.id AS ticket, summary\n  FROM ticket t,enum p\n  WHERE t.status IN ('new', 'assigned', 'reopened') \n    AND p.name=t.priority AND p.type='priority'\n  ORDER BY t.milestone, p.value, t.severity, t.time\n</pre><p>\n<strong>Note:</strong> A table join is used to match <em>ticket</em> priorities with their\nnumeric representation from the <em>enum</em> table.\n</p>\n<h3 id=\"column-syntax\">Changing layout of report rows</h3>\n<p>\nBy default, all columns on each row are display on a single row in the HTML\nreport, possibly formatted according to the descriptions above. However, it's\nalso possible to create multi-line report entries.\n</p>\n<ul><li><strong><tt>column_</tt></strong> \u2014 <em>Break row after this</em>. By appending an underscore ('_') to the column name, the remaining columns will be continued on a second line.\n</li></ul><ul><li><strong><tt>_column_</tt></strong> \u2014 <em>Full row</em>. By adding an underscore ('_') both at the beginning and the end of a column name, the data will be shown on a separate row.\n</li></ul><ul><li><strong><tt>_column</tt></strong> \u2014 <em>Hide data</em>. Prepending an underscore ('_') to a column name instructs Trac to hide the contents from the HTML output. This is useful for information to be visible only if downloaded in other formats (like CSV or RSS/XML).\nThis can be used to hide any kind of column, even important ones required for identifying the resource, e.g. <tt>id as _id</tt> will hide the <strong>Id</strong> column but the link to the ticket will be present.\n</li></ul><p>\n<strong>Example:</strong> <em>List active tickets, grouped by milestone, colored by priority, with  description and multi-line layout</em>\n</p>\n<pre class=\"wiki\">SELECT p.value AS __color__,\n       t.milestone AS __group__,\n       (CASE owner \n          WHEN 'daniel' THEN 'font-weight: bold; background: red;' \n          ELSE '' END) AS __style__,\n       t.id AS ticket, summary AS summary_,             -- ## Break line here\n       component,version, severity, milestone, status, owner,\n       time AS created, changetime AS modified,         -- ## Dates are formatted\n       description AS _description_,                    -- ## Uses a full row\n       changetime AS _changetime, reporter AS _reporter -- ## Hidden from HTML output\n  FROM ticket t,enum p\n  WHERE t.status IN ('new', 'assigned', 'reopened') \n    AND p.name=t.priority AND p.type='priority'\n  ORDER BY t.milestone, p.value, t.severity, t.time\n</pre><h3 id=\"Reportingoncustomfields\">Reporting on custom fields</h3>\n<p>\nIf you have added custom fields to your tickets (a feature since v0.8, see <a class=\"wiki\" href=\"/LUCICodeRepository/nomaticIM/wiki/TracTicketsCustomFields\">TracTicketsCustomFields</a>), you can write a SQL query to cover them. You'll need to make a join on the ticket_custom table, but this isn't especially easy.\n</p>\n<p>\nIf you have tickets in the database <em>before</em> you declare the extra fields in trac.ini, there will be no associated data in the ticket_custom table. To get around this, use SQL's \"LEFT OUTER JOIN\" clauses. See <a class=\"ext-link\" href=\"http://trac.edgewall.org/intertrac/TracIniReportCustomFieldSample\" title=\"TracIniReportCustomFieldSample in Trac project trac\"><span class=\"icon\">\u200b</span>TracIniReportCustomFieldSample</a> for some examples.\n</p>\n<p>\n<strong>Note that you need to set up permissions in order to see the buttons for adding or editing
