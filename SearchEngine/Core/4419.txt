id=\"content_title\">Inappropriate Linux Configurations</div>\n\n<blockquote>\n<div id=\"content_text\">\n\n<p>In many cases, vendor-supplied files have inappropriate configurations\n              which\nshould be changed to prevent others from obtaining unwanted access.\n          </p>\n<ul>\n<li> <code>/etc/hosts.equiv</code> and <code>$HOME/.rhosts</code>\n \n<ul>\n<li> Decide if the file <code>/etc/hosts.equiv</code> is \nreally\n                required on your system.&#160; If you are running commands \nsuch\n                as <code>rsh</code>, <code>rlogin</code>, etc., this \nfile allows\n                other hosts to be trusted by your system.&#160; Rlogin will \nthen\n                allow someone with the same account name to login to \nyour machine\n                from a listed host without supplying a password. This is\n a way\n                that hackers can easily leap-frog from machine to \nmachine. </li>\n<li> Verify that the permissions on these files are set to 600.</li>\n<li> Verify that the owner of <code>$HOME/.rhosts</code>&#160; \nis\n                the same as the account's owner and that the owner of <code>/etc/hosts.equiv</code>\n is <i>root</i>.</li>\n<li> Make sure that the file does NOT contain the symbol <b>\"+\"</b>\n on\n                any line as this allows anyone access to this account or\n system.</li>\n<li> Verify that the usage of netgroups within <code>.rhosts</code>\n does\n                not allow unintended access to this account. Make sure \nthat only\n                the accounts and hosts you want to provide access to \nhave access.</li>\n<li> Do not use <b>'!'</b> or <b>'#'</b> in these files.&#160; \nThere\n                are no comment characters for these files. </li>\n<li> Only trust hosts which are within your domain or under your direct managment. </li>\n<li> Only use fully qualified hostnames (i.e., <i>hostname.ics.uci.edu</i>).</li>\n</ul>\n</li>\n<p>&#160;</p>\n<li> <code>/usr/lib/X11/xdm/Xsession</code> </li>\n<p> Check this file for an <code>xhost</code> command with a\n <b>'+'</b>.&#160; Remove\n              that line, since it allows anyone on the network (or \npossibly on\n              the Internet) to talk to your X server, insert commands \ninto windows, and read your console keystrokes.            </p>\n<p>&#160;</p>\n<li> <code>/etc/ttys</code> and <code>/etc/ttytab</code></li>\n<p> The only terminal that should be set to <i>secure</i> \nshould\n              be the console.            </p>\n<p>&#160;</p>\n<li> <code>/etc/aliases</code> (or <code>/usr/lib/aliases</code>)</li>\n<p> Check this mail alias file for inappropriate entries. When shipped, some alias\n              files include an alias named <i>uudecode</i> or\n just <i>decode</i>.&#160; This\n              can almost always be commented out.            </p>\n<p>&#160;</p>\n<li> System files\n<ul>\n<li> Check the permissions and ownership of system files &#160;and directories,\n                especially the <b>/</b> (root) and <code>/etc</code> \ndirectories,\n                and all system and network configuration files. </li>\n<li> Examine file and directory protections before and \nafter installing\n                software. These procedures can cause file and directory \nprotections\n                to change without you being aware of it. </li>\n</ul>\n</li>\n<p>&#160;</p>\n<li> Setuid shell scripts are always potential security \nproblems and can not be made secure.
