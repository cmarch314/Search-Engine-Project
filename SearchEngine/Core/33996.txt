class=\"False\"><a class=\"False\" href=\"/LUCICodeRepository/nomaticIM/wiki/TracNotification\">Notification</a></li></ul></div><p>\n</p>\n<p>\nThe Trac ticket database provides simple but effective tracking of issues and bugs within a project.\n</p>\n<p>\nAs the central project management element of Trac, tickets are used for <strong>project tasks</strong>, <strong>feature requests</strong>, <strong>bug reports</strong> and <strong>software support issues</strong>. \n</p>\n<p>\nAs with the <a class=\"wiki\" href=\"/LUCICodeRepository/nomaticIM/wiki/TracWiki\">TracWiki</a>, this subsystem has been designed with the goal of making user contribution and participation as simple as possible. It should be as easy as possible to report bugs, ask questions and suggest improvements.\n</p>\n<p>\nAn issue is assigned to a person who must resolve it or reassign the ticket to someone else.\nAll tickets can be edited, annotated, assigned, prioritized and discussed at any time.\n</p>\n<h2 id=\"TicketFields\">Ticket Fields</h2>\n<p>\nA  ticket contains the following information attributes:\n \n</p>\n<ul><li><strong>Reporter</strong> \u2014 The author of the ticket.\n</li><li><strong>Type</strong> \u2014 The nature of the ticket (for example, defect or enhancement request)\n</li></ul><ul><li><strong>Component</strong> \u2014 The project module or subsystem this ticket concerns.\n</li><li><strong>Version</strong> \u2014 Version of the project that this ticket pertains to.\n</li><li><strong>Keywords</strong> \u2014 Keywords that a ticket is marked with.  Useful for searching and report generation.\n</li></ul><ul><li><strong>Priority</strong> \u2014 The importance of this issue, ranging from <em>trivial</em> to <em>blocker</em>.\n</li><li><strong>Milestone</strong> \u2014 When this issue should be resolved at the latest.\n</li><li><strong>Assigned to/Owner</strong> \u2014 Principal person responsible for handling the issue.\n</li><li><strong>Cc</strong> \u2014 A comma-separated list of other users or E-Mail addresses to notify. <em>Note that this does not imply responsiblity or any other policy.</em>\n</li></ul><p>\n \n</p>\n<ul><li><strong>Resolution</strong> \u2014 Reason for why a ticket was closed. One of <tt>fixed</tt>, <tt>invalid</tt>, <tt>wontfix</tt>, <tt>duplicate</tt>, <tt>worksforme</tt>.\n</li><li><strong>Status</strong> \u2014 What is the current status? One of <tt>new</tt>, <tt>assigned</tt>, <tt>closed</tt>, <tt>reopened</tt>.\n</li><li><strong>Summary</strong> \u2014 A brief description summarizing the problem or issue.\n</li><li><strong>Description</strong> \u2014 The body of the ticket. A good description should be specific, descriptive and to the point.\n</li></ul><p>\n<strong>Note:</strong> Versions of Trac prior to 0.9 did not have the <em>type</em> field, but instead provided a <em>severity</em> field and different default values for the <em>priority</em> field. This change was done to simplify the ticket model by removing the somewhat blurry distinction between <em>priority</em> and <em>severity</em>. However, the old model is still available if you prefer it: just add/modify the default values of the <em>priority</em> and <em>severity</em>, and optionally hide the <em>type</em> field by removing all the possible values through <a class=\"wiki\" href=\"/LUCICodeRepository/nomaticIM/wiki/TracAdmin\">trac-admin</a>.\n</p>\n<p>\n<strong>Note:</strong> the <a class=\"ext-link\" href=\"http://trac.edgewall.org/intertrac/TicketTypes\" title=\"TicketTypes in Trac project trac\"><span class=\"icon\">\u200b</span>type</a>, <a class=\"ext-link\" href=\"http://trac.edgewall.org/intertrac/TicketComponent\" title=\"TicketComponent in Trac project trac\"><span class=\"icon\">\u200b</span>component</a>, version, priority and severity fields can be managed with <a class=\"wiki\" href=\"/LUCICodeRepository/nomaticIM/wiki/TracAdmin\">trac-admin</a> or with the <a class=\"ext-link\" href=\"http://trac.edgewall.org/intertrac/WebAdmin\" title=\"WebAdmin in Trac project trac\"><span class=\"icon\">\u200b</span>WebAdmin</a> plugin.\n</p>\n<p>\n<strong>Note:</strong> Description of the builtin <em>priority</em> values is available at <a class=\"ext-link\" href=\"http://trac.edgewall.org/intertrac/TicketTypes%23Whyistheseverityfieldgone\" title=\"TicketTypes#Whyistheseverityfieldgone in Trac project trac\"><span class=\"icon\">\u200b</span>TicketTypes</a>\n</p>\n<h2 id=\"ChangingandCommentingTickets\">Changing and Commenting Tickets</h2>\n<p>\nOnce a ticket has been entered into Trac, you can at any time change the\ninformation by <strong>annotating</strong> the bug. This means changes and comments to\nthe ticket are logged as a part of the ticket itself.\n</p>\n<p>\nWhen viewing a ticket, the history of changes will appear below the main ticket area.\n</p>\n<p>\n<em>In the Trac project, we use ticket comments to discuss issues and tasks. This makes\nunderstanding the motivation behind a design- or implementation choice easier,\nwhen returning to it later.</em>\n</p>\n<p>\n<strong>Note:</strong> An important feature is being able to use <a class=\"wiki\" href=\"/LUCICodeRepository/nomaticIM/wiki/TracLinks\">TracLinks</a> and\n<a class=\"wiki\" href=\"/LUCICodeRepository/nomaticIM/wiki/WikiFormatting\">WikiFormatting</a> in ticket descriptions and comments. Use <a class=\"wiki\" href=\"/LUCICodeRepository/nomaticIM/wiki/TracLinks\">TracLinks</a> to refer to\nother issues, changesets or files to make your ticket more specific and easier\nto understand.\n</p>\n<p>\n<strong>Note:</strong> See <a class=\"wiki\" href=\"/LUCICodeRepository/nomaticIM/wiki/TracNotification\">TracNotification</a> for how to configure email notifications of ticket changes.\n</p>\n<p>\n<strong>Note:</strong> See <a class=\"wiki\" href=\"/LUCICodeRepository/nomaticIM/wiki/TracWorkflow\">TracWorkflow</a> for information about the state transitions (ticket lifecycle), and how this workflow can be customized.\n</p>\n<h2 id=\"DefaultValuesforDrop-DownFields\">Default Values for Drop-Down Fields</h2>\n<p>\nThe option selected by default for the various drop-down fields can be set in <a class=\"wiki\" href=\"/LUCICodeRepository/nomaticIM/wiki/TracIni\">trac.ini</a>, in the <tt>[ticket]</tt> section:\n</p>\n<ul><li><tt>default_component</tt>: Name of the component selected by default\n</li><li><tt>default_milestone</tt>: Name of the default milestone\n</li><li><tt>default_priority</tt>: Default priority value\n</li><li><tt>default_severity</tt>: Default severity value\n</li><li><tt>default_type</tt>: Default ticket type\n</li><li><tt>default_version</tt>: Name of the default version\n</li><li><tt>default_owner</tt>: Name of the default owner, <em>if no owner for the component has been set</em>\n</li></ul><p>\nIf any of these options are omitted, the default value will either be the first in the list, or an empty value, depending on whether the field in question is required to be set.  Some of these can be chosen through the <a class=\"ext-link\" href=\"http://trac.edgewall.org/intertrac/WebAdmin\" title=\"WebAdmin in Trac project trac\"><span class=\"icon\">\u200b</span>WebAdmin</a> plugin in the \"Ticket System\" section (others in the \"trac.ini\" section).  The default owner for a ticket will be the component owner, if that is set, or <tt>default_owner</tt>, if not.\n</p>\n<h2 id=\"HidingFieldsandAddingCustomFields\">Hiding Fields and Adding Custom Fields</h2>\n<p>\nMany of the default ticket fields can be hidden from the ticket web interface simply by removing all the possible values through <a class=\"wiki\" href=\"/LUCICodeRepository/nomaticIM/wiki/TracAdmin\">trac-admin</a>. This of course only applies to drop-down fields, such as <em>type</em>, <em>priority</em>, <em>severity</em>, <em>component</em>, <em>version</em> and <em>milestone</em>.\n</p>\n<p>\nTrac also lets you add your own custom ticket fields. See <a class=\"wiki\" href=\"/LUCICodeRepository/nomaticIM/wiki/TracTicketsCustomFields\">TracTicketsCustomFields</a> for more information.\n</p>\n<h2 id=\"Assign-toasDrop-DownList\">Assign-to as Drop-Down List</h2>\n<p>\nIf the list of possible ticket owners is finite, you can change the <em>assign-to</em> ticket field from a text input to a drop-down list. This is done by setting the <tt>restrict_owner</tt> option of the <tt>[ticket]</tt> section in <a class=\"wiki\" href=\"/LUCICodeRepository/nomaticIM/wiki/TracIni\">trac.ini</a> to \u201ctrue\u201d. In that case, Trac will use the list of all users who have accessed the project to populate the drop-down field.\n</p>\n<p>\nTo appear in the dropdown list, a user needs be registered with the project, <em>i.e.</em> a user session should exist in the database. Such an entry is automatically created in the database the first time the user submits a change in the project, for example when editing the user's details in the <em>Settings</em> page, or simply by authenticating if the user has a login. Also, the user must have <tt>TICKET_MODIFY</tt> <a class=\"wiki\" href=\"/LUCICodeRepository/nomaticIM/wiki/TracPermissions\">permissions</a>.\n</p>\n<p>\n<strong>Note:</strong> See <a class=\"ext-link\" href=\"http://pacopablo.com/wiki/pacopablo/blog/set-assign-to-drop-down\"><span class=\"icon\">\u200b</span>Populating Assign To Drop Down</a> on how to add user entries at database level\n</p>\n<p>\n<strong>Note 2:</strong> If you need serious flexibility and aren't afraid of a little plugin coding of your own, see <a class=\"ext-link\" href=\"http://trac-hacks.org/wiki/FlexibleAssignToPlugin\"><span class=\"icon\">\u200b</span>FlexibleAssignTo</a> (disclosure: I'm the author)\n</p>\n<p>\n<strong>Note 3:</strong> Activating this option may cause some performance degradation, read more about this in the <a class=\"ext-link\" href=\"http://trac.edgewall.org/intertrac/TracPerformance%23Configuration\" title=\"TracPerformance#Configuration in Trac project trac\"><span class=\"icon\">\u200b</span>Trac performance</a> page.\n</p>\n<h2 id=\"PresetValuesforNewTickets\">Preset Values for New Tickets</h2>\n<p>\nTo create a link to the new-ticket form filled with preset values, you need to call the <tt>/newticket?</tt> URL with variable=value separated by &amp;. \n</p>\n<p>\nPossible variables are :\n</p>\n<ul><li><strong>type</strong> \u2014 The type droplist\n</li><li><strong>reporter</strong> \u2014 Name or email of the reporter\n</li><li><strong>summary</strong> \u2014 Summary line for the ticket\n</li><li><strong>description</strong> \u2014 Long description of the ticket\n</li><li><strong>component</strong> \u2014 The component droplist\n</li><li><strong>version</strong> \u2014 The version droplist\n</li><li><strong>severity</strong> \u2014 The severity droplist\n</li><li><strong>keywords</strong> \u2014 The keywords \n</li><li><strong>priority</strong> \u2014 The priority droplist\n</li><li><strong>milestone</strong> \u2014 The milestone droplist\n</li><li><strong>owner</strong> \u2014 The person responsible for the ticket\n</li><li><strong>cc</strong> \u2014 The list of emails for notifying about the ticket change\n</li></ul><p>\n<strong>Example:</strong> <em>/trac/newticket?summary=Compile%20Error&amp;version=1.0&amp;component=gui</em><br />\n</p>\n<hr
