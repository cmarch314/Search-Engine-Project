id=\"Finegrainedpermissions\">Fine grained permissions</h1>\n<p>\nBefore Trac 0.11, it was only possible to define fine-grained permissions checks on the repository browser sub-system.\n</p>\n<p>\nSince 0.11, there's a general mechanism in place that allows custom <strong>permission policy plugins</strong> to grant or deny any action on any kind of Trac resources, even at the level of specific versions of such resources.\n</p>\n<p>\nNote that for Trac 0.12, <tt>authz_policy</tt> has been integrated as an optional module (in <tt>tracopt.perm.authz_policy.*</tt>), so it's installed by default and can simply be activated via the <em>Plugins</em> panel in the Trac administration module.\n</p>\n<h2 id=\"PermissionPolicies\">Permission Policies</h2>\n<p>\nA great diversity of permission policies can be implemented, and Trac comes with a few examples. \n</p>\n<p>\nWhich policies are currently active is determined by a configuration setting in <a class=\"wiki\" href=\"/LUCICodeRepository/nomaticIM/wiki/TracIni\">TracIni</a>:\ne.g.\n</p>\n<pre class=\"wiki\">[trac]\npermission_policies = AuthzSourcePolicy, DefaultPermissionPolicy, LegacyAttachmentPolicy\n</pre><p>\nThis lists the <a class=\"wiki\" href=\"/LUCICodeRepository/nomaticIM/wiki/TracFineGrainedPermissions#AuthzSourcePolicy\">#AuthzSourcePolicy</a> described below as the first policy, followed by the DefaultPermissionPolicy which checks for the traditional coarse grained style permissions described in <a class=\"wiki\" href=\"/LUCICodeRepository/nomaticIM/wiki/TracPermissions\">TracPermissions</a>, and the LegacyAttachmentPolicy which knows how to use the coarse grained permissions for checking the permissions available on attachments.\n</p>\n<p>\nAmong the possible optional choices, there is <a class=\"wiki\" href=\"/LUCICodeRepository/nomaticIM/wiki/TracFineGrainedPermissions#AuthzPolicy\">#AuthzPolicy</a>, a very generic permission policy, based on an Authz-style system. See\n<a class=\"ext-link\" href=\"http://trac.edgewall.org/intertrac/source%3Abranches/0.12-stable/tracopt/perm/authz_policy.py\" title=\"source:branches/0.12-stable/tracopt/perm/authz_policy.py in Trac project trac\"><span class=\"icon\">\u200b</span>authz_policy.py</a> for details. \n</p>\n<p>\nAnother popular permission policy <a class=\"wiki\" href=\"/LUCICodeRepository/nomaticIM/wiki/TracFineGrainedPermissions#AuthzSourcePolicy\">#AuthzSourcePolicy</a>, re-implements the pre-0.12 support for checking fine-grained permissions limited to Subversion repositories in terms of the new system.\n</p>\n<p>\nSee also <a class=\"ext-link\" href=\"http://trac.edgewall.org/intertrac/source%3Abranches/0.12-stable/sample-plugins/permissions\" title=\"source:branches/0.12-stable/sample-plugins/permissions in Trac project trac\"><span class=\"icon\">\u200b</span>sample-plugins/permissions</a> for more examples.\n</p>\n<h3 id=\"AuthzPolicy\">AuthzPolicy</h3>\n<ul><li>Install <a class=\"ext-link\" href=\"http://www.voidspace.org.uk/python/configobj.html\"><span class=\"icon\">\u200b</span>ConfigObj</a> (required).\n</li><li>Copy authz_policy.py into your plugins directory.\n</li><li>Put a <a class=\"ext-link\" href=\"http://swapoff.org/files/authzpolicy.conf\"><span class=\"icon\">\u200b</span>authzpolicy.conf</a> file somewhere, preferably on a secured location on the server, not readable for others than the webuser. If the  file contains non-ASCII characters, the UTF-8 encoding should be used.\n</li><li>Update your <tt>trac.ini</tt>:\n<ol><li>modify the <a class=\"wiki\" href=\"/LUCICodeRepository/nomaticIM/wiki/TracIni#trac-section\">permission_policies</a> entry in the <tt>[trac]</tt> section\n<pre class=\"wiki\">[trac]\n...\npermission_policies = AuthzPolicy, DefaultPermissionPolicy, LegacyAttachmentPolicy\n</pre></li><li>add a new <tt>[authz_policy]</tt> section\n<pre class=\"wiki\">[authz_policy]\nauthz_file = /some/trac/env/conf/authzpolicy.conf\n</pre></li><li>enable the single file plugin\n<pre class=\"wiki\">[components]\n...\n# Trac 0.12\ntracopt.perm.authz_policy.* = enabled\n# for Trac 0.11 use this\n#authz_policy.* = enabled \n</pre></li></ol></li></ul><p>\nNote that the order in which permission policies are specified is quite critical, \nas policies will be examined in the sequence provided.\n</p>\n<p>\nA policy will return either <tt>True</tt>, <tt>False</tt> or <tt>None</tt> for a given permission check.\nOnly if the return value is <tt>None</tt> will the <em>next</em> permission policy be consulted.\nIf no policy explicitly grants the permission, the final result will be <tt>False</tt> \n(i.e. no permission).\n</p>\n<p>\nFor example, if the <tt>authz_file</tt> contains:\n</p>\n<pre class=\"wiki\">[wiki:WikiStart@*]\n* = WIKI_VIEW\n\n[wiki:PrivatePage@*]\njohn = WIKI_VIEW\n* =\n</pre><p>\nand the default permissions are set like this:\n</p>\n<pre class=\"wiki\">john           WIKI_VIEW\njack           WIKI_VIEW\n# anonymous has no WIKI_VIEW\n</pre><p>\nThen: \n</p>\n<ul><li>All versions of <a class=\"wiki\" href=\"/LUCICodeRepository/nomaticIM/wiki/WikiStart\">WikiStart</a> will be viewable by everybody (including anonymous)\n</li><li>PrivatePage will be viewable only by john\n</li><li>other pages will be viewable only by john and jack\n</li></ul><h3 id=\"AuthzSourcePolicy\">AuthzSourcePolicy  (mod_authz_svn-like permission policy)</h3>\n<p>\nAt the time of this writing, the old fine grained permissions system from Trac 0.11 and before used for restricting access to the repository has  been converted to a permission policy component, but from the user point of view, this makes little if no difference.\n</p>\n<p>\nThat kind of fine-grained permission control needs a definition file, which is the one used by Subversion's mod_authz_svn. \nMore information about this file format and about its usage in Subversion is available in the  <a class=\"ext-link\" href=\"http://svnbook.red-bean.com/en/1.5/svn.serverconfig.pathbasedauthz.html\"><span class=\"icon\">\u200b</span>Path-Based Authorization</a> section in the Server Configuration chapter of the svn book.\n</p>\n<p>\nExample:\n</p>\n<pre class=\"wiki\">[/]\n* = r\n\n[/branches/calc/bug-142]\nharry = rw\nsally = r\n\n[/branches/calc/bug-142/secret]\nharry =\n</pre><ul><li><strong>/</strong> = <em>Everyone has read access by default</em>\n</li><li><strong>/branches/calc/bug-142</strong> = <em>harry has read/write access, sally read only</em>\n</li><li><strong>/branches/calc/bug-142/secret</strong> = <em>harry has no access, sally has read access (inherited as a sub folder permission)</em>\n</li></ul><h4 id=\"TracConfiguration\">Trac Configuration</h4>\n<p>\nTo activate fine grained permissions you <span class=\"underline\">must</span> specify the <tt>authz_file</tt> option in the <tt>[trac]</tt> section of trac.ini. If this option is set to null or not specified the permissions will not be used.\n</p>\n<pre class=\"wiki\">[trac]\nauthz_file = /path/to/svnaccessfile\n</pre><p>\nIf you want to support the use of the <tt>[</tt><em>modulename</em><tt>:/</tt><em>some</em><tt>/</tt><em>path</em><tt>]</tt> syntax within the <tt>authz_file</tt>, add \n</p>\n<pre class=\"wiki\">authz_module_name = modulename\n</pre><p>\nwhere <em>modulename</em> refers to the same repository indicated by the <tt>repository_dir</tt> entry in the <tt>[trac]</tt> section. As an example, if the <tt>repository_dir</tt> entry in the <tt>[trac]</tt> section is <tt>/srv/active/svn/blahblah</tt>, that would yield the following:\n</p>\n<pre class=\"wiki\">[trac]\nauthz_file = /path/to/svnaccessfile\nauthz_module_name = blahblah\n...\nrepository_dir = /srv/active/svn/blahblah \n</pre><p>\nwhere the svn access file, <tt>/path/to/svnaccessfile</tt>, contains entries such as <tt>[blahblah:/some/path]</tt>.\n</p>\n<p>\n<strong>Note:</strong> Usernames inside the Authz file <span class=\"underline\">must</span> be the same as those used inside trac. \n</p>\n<p>\nAs of version 0.12, make sure you have <em>AuthzSourcePolicy</em> included in the permission_policies list in trac.ini, otherwise the authz permissions file will be ignored.\n</p>\n<pre class=\"wiki\">[trac]\npermission_policies = AuthzSourcePolicy, DefaultPermissionPolicy, LegacyAttachmentPolicy\n</pre><h4 id=\"SubversionConfiguration\">Subversion Configuration</h4>\n<p>\nThe same access file is typically applied to the corresponding Subversion repository using an Apache directive like this:\n</p>\n<pre class=\"wiki\">&lt;Location /repos&gt;\n  DAV svn\n  SVNParentPath /usr/local/svn\n\n  # our access control policy\n  AuthzSVNAccessFile /path/to/svnaccessfile\n&lt;/Location&gt;\n</pre><p>\nFor information about how to restrict access to entire projects in a multiple project environment see <a class=\"ext-link\" href=\"http://trac.edgewall.org/intertrac/wiki%3ATracMultipleProjectsSVNAccess\" title=\"wiki:TracMultipleProjectsSVNAccess in Trac project trac\"><span class=\"icon\">\u200b</span>wiki:TracMultipleProjectsSVNAccess</a>\n</p>\n<h2 id=\"Debuggingpermissions\">Debugging permissions</h2>\n<p>\nIn trac.ini set:\n</p>\n<pre class=\"wiki\">[logging]\nlog_file = trac.log\nlog_level = DEBUG\nlog_type = file\n</pre><p>\nAnd watch:\n</p>\n<pre class=\"wiki\">tail -n 0 -f log/trac.log | egrep '\\[perm\\]|\\[authz_policy\\]'\n</pre><p>\nto understand what checks are being performed. See the sourced documentation of the plugin for more info.\n</p>\n<hr />\n<p>\nSee also: <a class=\"wiki\" href=\"/LUCICodeRepository/nomaticIM/wiki/TracPermissions\">TracPermissions</a>,\n<a
