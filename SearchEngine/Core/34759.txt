class=\"False\"><a class=\"False\" href=\"/LUCICodeRepository/nomaticIM/wiki/TracNotification\">Notification</a></li></ul></div><p>\n</p>\n<p>\nTrac uses a simple, case sensitive, permission system to control what users can and can't access.\n</p>\n<p>\nPermission privileges are managed using the <a class=\"wiki\" href=\"/LUCICodeRepository/nomaticIM/wiki/TracAdmin\">trac-admin</a> tool or (new in version 0.11) the <em>General / Permissions</em> panel in the <em>Admin</em> tab of the web interface.\n</p>\n<p>\nIn addition to the default permission policy described in this page, it is possible to activate additional permission policies by enabling plugins and listing them in the <tt>[trac] permission_policies</tt> configuration entry in the <a class=\"wiki\" href=\"/LUCICodeRepository/nomaticIM/wiki/TracIni\">TracIni</a>. See <a class=\"wiki\" href=\"/LUCICodeRepository/nomaticIM/wiki/TracFineGrainedPermissions\">TracFineGrainedPermissions</a> for more details.\n</p>\n<p>\nNon-authenticated users accessing the system are assigned the name \"anonymous\". Assign permissions to the \"anonymous\" user to set privileges for anonymous/guest users. The parts of Trac that a user does not have the privileges for will not be displayed in the navigation.\nIn addition to these privileges, users can be granted additional individual rights in effect when authenticated and logged into the system. All logged in users belong to the virtual group \"authenticated\", which inherits permissions from \"anonymous\".\n</p>\n<h2 id=\"GraphicalAdminTab\">Graphical Admin Tab</h2>\n<p>\n<em>This feature is new in version 0.11.</em>\n</p>\n<p>\nTo access this tab, a user must have <tt>TRAC_ADMIN privileges</tt>. This can be performed as follows (more on the trac-admin script below):\n</p>\n<pre class=\"wiki\">  $ trac-admin /path/to/projenv permission add bob TRAC_ADMIN\n</pre><p>\nThen, the user <tt>bob</tt> will be able to see the Admin tab, and can then access the permissions menu. This menu will allow you to perform all the following actions, but from the browser without requiring root access to the server (just the correct permissions for your user account).\u2003  <strong>Use at least one lowercase character in user names, as all-uppercase names are reserved for permissions.</strong>\n</p>\n<ol><li><a style=\"padding:0; border:none\" href=\"/LUCICodeRepository/nomaticIM/chrome/site/../common/guide/admin.png\"><img src=\"/LUCICodeRepository/nomaticIM/chrome/site/../common/guide/admin.png\" alt=\"admin.png\" title=\"admin.png\" /></a>\n</li><li><a style=\"padding:0; border:none\" href=\"/LUCICodeRepository/nomaticIM/chrome/site/../common/guide/admin-permissions.png\"><img src=\"/LUCICodeRepository/nomaticIM/chrome/site/../common/guide/admin-permissions.png\" alt=\"admin-permissions.png\" title=\"admin-permissions.png\" /></a>\n</li><li><a style=\"padding:0; border:none\" href=\"/LUCICodeRepository/nomaticIM/chrome/site/../common/guide/admin-permissions-TICKET_ADMIN.png\"><img src=\"/LUCICodeRepository/nomaticIM/chrome/site/../common/guide/admin-permissions-TICKET_ADMIN.png\" alt=\"admin-permissions-TICKET_ADMIN.png\" title=\"admin-permissions-TICKET_ADMIN.png\" /></a>\n</li></ol><p>\nAn easy way to quickly secure a new Trac install is to run the above command on the anonymous user, install the <a class=\"ext-link\" href=\"http://trac-hacks.org/wiki/AccountManagerPlugin\"><span class=\"icon\">\u200b</span>AccountManagerPlugin</a>, create a new admin account graphically and then remove the TRAC_ADMIN permission from the anonymous user.\n</p>\n<h2 id=\"AvailablePrivileges\">Available Privileges</h2>\n<p>\nTo enable all privileges for a user, use the <tt>TRAC_ADMIN</tt> permission. permissions, for example to be used in a custom workflow, enable the optional <tt>tracopt.perm.config_perm_provider.ExtraPermissionsProvider</tt> component in the \"Plugins\" admin panel, and add the desired permissions to the <tt>[extra-permissions]</tt> section in your <a class=\"wiki\" href=\"/LUCICodeRepository/nomaticIM/wiki/TracIni#extra-permissions-section\">trac.ini</a>. For more information, please refer to the documentation of the component in the admin panel.\n</p>\n<h2 id=\"GrantingPrivileges\">Granting Privileges</h2>\n<p>\nYou grant privileges to users using <a class=\"wiki\" href=\"/LUCICodeRepository/nomaticIM/wiki/TracAdmin\">trac-admin</a>. The current set of privileges can be listed with the following command:\n</p>\n<pre class=\"wiki\">  $ trac-admin /path/to/projenv permission list\n</pre><p>\nThis command will allow the user <em>bob</em> to delete reports:\n</p>\n<pre class=\"wiki\">  $ trac-admin /path/to/projenv permission add bob REPORT_DELETE\n</pre><p>\nThe <tt>permission add</tt> command also accepts multiple privilege names:\n</p>\n<pre class=\"wiki\">  $ trac-admin /path/to/projenv permission add bob REPORT_DELETE WIKI_CREATE\n</pre><p>\nOr add all privileges:\n</p>\n<pre class=\"wiki\">  $ trac-admin /path/to/projenv permission add bob TRAC_ADMIN\n</pre><h2 id=\"PermissionGroups\">Permission Groups</h2>\n<p>\nThere are two built-in groups, \"authenticated\" and \"anonymous\".\nAny user who has not logged in is automatically in the \"anonymous\" group.\nAny user who has logged in is also in the \"authenticated\" group.\nThe \"authenticated\" group inherits permissions from the \"anonymous\" group.\nFor example, if the \"anonymous\" group has permission WIKI_MODIFY, \nit is not necessary to add the WIKI_MODIFY permission to the \"authenticated\" group as well.\n</p>\n<p>\nCustom groups may be defined that inherit permissions from the two built-in groups.\n</p>\n<p>\nPermissions can be grouped together to form roles such as <em>developer</em>, <em>admin</em>, etc.\n</p>\n<pre class=\"wiki\">  $ trac-admin /path/to/projenv permission add developer WIKI_ADMIN\n  $ trac-admin /path/to/projenv permission add developer REPORT_ADMIN\n  $ trac-admin /path/to/projenv permission add developer TICKET_MODIFY\n  $ trac-admin /path/to/projenv permission add bob developer\n  $ trac-admin /path/to/projenv permission add john developer\n</pre><p>\nGroup membership can be checked by doing a <tt>permission list</tt> with no further arguments; the resulting output will include group memberships. <strong>Use at least one lowercase character in group names, as all-uppercase names are reserved for permissions</strong>.\n</p>\n<h2 id=\"AddingaNewGroupandPermissions\">Adding a New Group and Permissions</h2>\n<p>\nPermission groups can be created by assigning a user to a group you wish to create, then assign permissions to that group.\n</p>\n<p>\nThe following will add <em>bob</em> to the new group called <em>beta_testers</em> and then will assign WIKI_ADMIN permissions to that group. (Thus, <em>bob</em> will inherit the WIKI_ADMIN permission)\n</p>\n<pre class=\"wiki\">   $ trac-admin /path/to/projenv permission add bob beta_testers\n   $ trac-admin /path/to/projenv permission add beta_testers WIKI_ADMIN\n\n</pre><h2 id=\"RemovingPermissions\">Removing Permissions</h2>\n<p>\nPermissions can be removed using the 'remove' command. For example:\n</p>\n<p>\nThis command will prevent the user <em>bob</em> from deleting reports:\n</p>\n<pre class=\"wiki\">  $ trac-admin /path/to/projenv permission remove bob REPORT_DELETE\n</pre><p>\nJust like <tt>permission add</tt>, this command accepts multiple privilege names.\n</p>\n<p>\nYou can also remove all privileges for a specific user:\n</p>\n<pre class=\"wiki\">  $ trac-admin /path/to/projenv permission remove bob '*'\n</pre><p>\nOr one privilege for all users:\n</p>\n<pre class=\"wiki\">  $ trac-admin /path/to/projenv permission remove '*' REPORT_ADMIN\n</pre><h2 id=\"DefaultPermissions\">Default Permissions</h2>\n<p>\nBy default on a new Trac installation, the <tt>anonymous</tt> user will have <em>view</em> access to everything in Trac, but will not be able to create or modify anything.\nOn the other hand, the <tt>authenticated</tt> users will have the permissions to <em>create and modify tickets and wiki pages</em>.\n</p>\n<p>\n<strong>anonymous</strong>\n</p>\n<pre class=\"wiki\"> BROWSER_VIEW \n CHANGESET_VIEW \n FILE_VIEW \n LOG_VIEW \n MILESTONE_VIEW \n REPORT_SQL_VIEW \n REPORT_VIEW \n ROADMAP_VIEW \n SEARCH_VIEW \n TICKET_VIEW \n TIMELINE_VIEW\n WIKI_VIEW\n</pre><p>\n<strong>authenticated</strong>\n</p>\n<pre class=\"wiki\"> TICKET_CREATE \n TICKET_MODIFY \n WIKI_CREATE \n
